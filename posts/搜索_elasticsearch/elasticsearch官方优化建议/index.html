二、基本建议

2.1 不要返回大数据集

elasticsearch设计为搜索引擎，可以获取匹配查询条件的文档集合, 但是如果查询的数据集合太大，例如：特定查询匹配全部的索引文档，效果会比较差

4.1 更多的内存分配给文件系统的缓存

elasticsearch 严重依赖于文件系统的缓存，用于加速搜索，通常情况下，保证有一半的物理内存用于文件系统的缓存，ES可以将热查询区域保存于物理内存中

4.3 文档建模

文档建模，可以更好的提高查询效率，

（1）避免使用join操作，嵌套nested 会使得查询速度减慢数倍，父子关系查询将会减慢数百倍

4.4 查询更少的字段

更多的字段查询，相比于单一字段查询，会更加慢，可以通过copy_to方式，再不修改源字段的情况下，添加新的查询字段（合并多个字段）

4.5 利用查询语句反向优化索引模版

例如：

(1) 入库的文档结构如下

PUT index/_doc/1
{
  "designation": "spoon",
  "price": 13
}

(2) 查询的请求如下

GET index/_search
{
  "aggs": {
    "price_ranges": {
      "range": {
        "field": "price",
        "ranges": [
          { "to": 10 },
          { "from": 10, "to": 100 },
          { "from": 100 }
        ]
      }
    }
  }
}

(3) 通过上面的分析，可以在入库时候，新增一个字段price_range, 聚合的语句将更快被执行

PUT index
{
  "mappings": {
    "properties": {
      "price_range": {
        "type": "keyword"
      }
    }
  }
}

PUT index/_doc/1
{
  "designation": "spoon",
  "price": 13,
  "price_range": "10-100"
}

GET index/_search
{
  "aggs": {
    "price_ranges": {
      "terms": {
        "field": "price_range"
      }
    }
  }
}

4.6 优先使用keyword字段，相比较于nunber类型

number类型（long， integer）的字段，执行范围查询比keyword快，但是，对于具体的term查询，keyword执行效率更高。

在下面的场景下，优先使用keyword类型:

(1) 不打算对于数字字段，进行范围查询； (2) 数字字段的快速查询非常重要

当然，也可以使用multi-field属性，生成同一个字段的多种类型，用于不同的查询需求

PUT yz_test/
{
  "mappings": {
    "properties": {
      "f1": {
        "type": "long",
        "fields": {
          "keyword": {
            "type": "keyword"
          }
        }
      }
    }
  }
}


POST yz_test/_doc
{
  "f1": 1000
}


GET yz_test/_search
{
  "query": {
    "term": {
      "f1.keyword": "1000"
    }
  }
}
